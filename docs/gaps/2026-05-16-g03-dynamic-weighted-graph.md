# G03 Dynamic Weighted Graph

Status: Blocking; current graph paths are live-data backed and now have a two-stage storage contract, but final full-corpus/browser QA is still open

## Requirement

ASIP must provide weighted relationship graph output. The user explicitly wants a global graph comparable to an Obsidian wiki relation graph, with connection weight reflected in the rendering.

SQLite is the persistent graph store. NetworkX is the in-memory graph runtime for traversal, subgraph extraction, and metrics.

Scope note: the original MVP-1 design deferred a full interactive graph canvas until the evidence workbench stabilized. The later active user request supersedes that deferral for this branch: a global weighted graph is now required for completion, while query/inspector views can still use bounded subgraphs.

Frontend rendering requirement: the Web graph must use a maintained React/npm graph visualization package instead of a hand-rolled SVG/canvas graph component. The package choice should be recorded before implementation. Candidate packages checked on 2026-05-17:

- `react-force-graph-2d` 1.29.1: React force-directed 2D graph; best fit for Obsidian-like global relation graph.
- `sigma` 3.0.3 with `graphology`: graph visualization for larger node/edge sets.
- `@xyflow/react` 12.10.2: best for node editors/flow charts, less natural for an Obsidian-style weighted relation graph.

Current preference for `/graph`: `react-force-graph-2d`, unless a quick spike proves Sigma is a better fit for density/performance.

Completeness requirement added after 2026-05-17 user review: a true global graph is not only the persisted `edges` table. The graph must merge at least these edge families from indexed data:

- persisted explicit graph edges generated by resolvers or semantic-edge jobs,
- evidence-derived symbol/source and symbol/symbol co-occurrence edges,
- code function operation edges from enclosing functions to registers they read/write/set; resolver wrapper names, macro names, expanded resolver calls, and field-only symbols stay in provenance/attributes and are not graph entity nodes,
- document/PDF/Markdown section nodes and section-to-symbol edges after conversion/extraction,
- batch LLM semantic edges generated from indexed code/doc/PDF/register candidates, with provenance and job status.

The first useful ASIP graph should therefore connect functions, registers, document sections, document boxes, and semantic relationships. A small disconnected register-only graph is not accepted as the global graph, but macro/function-like resolver helpers and field-only nodes are also not accepted because they create noisy mega-node structures.

Clarified hard boundary from 2026-05-17 user review: graph construction has at least two distinct stages, and completion must prove both separately.

Stage 1 is the deterministic code graph. For C/C++/kernel/MxGPU sources, this must be built from a code-aware pipeline such as clang/libclang/AST plus preprocessing or compile-command style macro expansion, not from regex-only resolver scans. This stage owns function and register nodes plus deterministic operation edges such as `reads`, `writes`, `sets_field`, `read_modify_writes`, and `maps_base`. YAML resolver profiles seed wrapper/macro interpretation, but wrapper names such as `WREG32`, `REG_SET_FIELD`, `SOC15_REG_OFFSET`, and `gpu_register` are resolver operations/provenance, not graph entity nodes.

Stage 1 also needs enough C call-graph structure to avoid a global graph made of disconnected register-operation islands. At minimum, source indexing must connect direct helper calls and AMD-style ops/vtable callback slots such as `.hw_init = gfx_v11_0_hw_init` with common dispatchers such as `funcs->hw_init(...)`. The callback slot/table names remain provenance only; visible graph nodes remain functions, registers, and document entities.

Stage 2 is LLM semantic-edge generation. It runs after Stage 1 and after document/PDF Markdown section extraction. It proposes additional semantic edges between already indexed code/doc/register concepts, records provider/model/provenance/job status, and must be distinguishable from deterministic edges in storage and QA. The LLM is not allowed to substitute for deterministic function/register extraction.

## Graph Entity Contract

The default graph is intentionally a compact BoxMatrix-style entity graph. Allowed node kinds are:

- `function`
- `register`
- `doc_section`
- `pdf_section`
- `doc_box`

Disallowed graph node kinds include resolver wrapper macros, intermediate macro-expansion helpers, field-only symbols, raw source file path nodes, temporary variables, and provider/model/config names. Those details remain available through edge provenance and node attributes.

Every graph node payload should have this shape:

```json
{
  "id": "stable canonical entity id",
  "kind": "function|register|doc_section|pdf_section|doc_box",
  "label": "human readable label",
  "in": ["incoming semantic or operation summaries"],
  "out": ["outgoing semantic or operation summaries"],
  "attr": {
    "source": [{"corpus_id": "mxgpu", "repo": "https://github.com/amd/MxGPU-Virtualization", "path": "relative/source/path", "line_start": 1, "line_end": 1}],
    "fields": ["field names touched by this register or operation"],
    "resolver_wrappers": ["provenance-only wrapper names"],
    "ip": "GC",
    "ip_version": "11.0",
    "asic": "navi3x",
    "summary": "doc box or section summary"
  }
}
```

Fields are attributes, not nodes. A `REG_SET_FIELD(tmp, GCVM_L2_CNTL, ENABLE_L2_CACHE, 1)` style operation should create or reinforce a function-to-register edge and record `ENABLE_L2_CACHE` in `attr.fields` or edge provenance, not create a standalone `ENABLE_L2_CACHE` node. A project-local macro that expands into `WREG32` or another resolver wrapper follows the same rule: the macro and wrapper stay in provenance, while the visible node is the function/register/doc entity.

Field-operation link invariant: if a function operates on a register field, the product graph must contain a traceable link from that function to the owning register. This can be a direct edge such as `function -> register` with relation `sets_field`, plus edge/node attributes recording the field, or an equivalent `function.out[]` entry that names the register and field. It is not valid to drop the field operation just because field nodes are hidden.

Required top-level node fields:

- `id`: stable canonical id used for edges.
- `kind`: one of the allowed node kinds.
- `label`: user-facing label.
- `in`: list of incoming relation summaries or empty list.
- `out`: list of outgoing relation summaries or empty list.
- `attr`: object containing required and optional typed attributes.

Required `attr` fields for all nodes:

- `source`: non-empty list of source records. Source records must include `corpus_id`, `repo`, and `path` when known. `line_start`, `line_end`, `page`, `commit`, and `snippet_id` are optional but should be preserved when available.

Additional required `attr` fields by kind:

- `function`: `function_name`, `language`, and `source`.
- `register`: `symbol`, `ip`, `ip_version`, and `source`. `ip_version` may be `unknown` only when extraction cannot prove it.
- `doc_section`/`pdf_section`: `source`, `section_id` or `anchor`, and `title`/`label`.
- `doc_box`: `source`, `box_id`, `summary`, `inputs`, `outputs`, and `constraints`.

Node merge rules:

- Nodes merge only when their canonical merge key is identical.
- `function` merge key: `function:<repo_or_corpus>:<normalized_path>:<function_name>`. Same function name in two files or two repos is not merged.
- `register` merge key: `register:<ip>:<ip_version>:<symbol>`. The same register symbol from two repos can merge if IP and IP version match. Different IP versions must not merge, even when the symbol is identical.
- `doc_section`/`pdf_section` merge key: `doc_section:<repo_or_corpus>:<path>:<anchor_or_page>`.
- `doc_box` merge key: `doc_box:<repo_or_corpus>:<path>:<box_id>`.
- When nodes merge, attributes are unioned rather than overwritten: `source`, `fields`, `resolver_wrappers`, `inputs`, `outputs`, and `constraints` are de-duplicated lists; scalar conflicts are kept in `attr.conflicts` unless one side is empty or `unknown`.
- Field operations are folded upward during merge: if two source edges mention `ENABLE_L2_CACHE` on `GCVM_L2_CNTL`, the merged register node keeps one `fields` entry and the merged function node keeps an `out` entry linking the operation to `register:GC:<ip_version>:GCVM_L2_CNTL`.

## Edge Enum Contract

Edges must also be normalized before product graph output. Free-form provider text or resolver access names are not allowed to leak into product relation names.

Allowed `edge.relation` enum:

- `reads`: function reads a register.
- `writes`: function writes a register.
- `sets_field`: function writes or updates one or more register fields; `attr.fields` must identify the field names when known.
- `maps_base`: function/register logic maps a register base or offset.
- `calls`: function calls another allowed function node.
- `contains`: document section contains a doc box.
- `documents`: doc section or doc box documents a function/register/doc box.
- `relates_to`: weak semantic relationship when no stricter enum applies.
- `depends_on`: source entity depends on target entity.
- `configures`: source configures target behavior/register.
- `resets`: source resets target hardware/software state.

Required edge fields:

- `src`: canonical allowed node id.
- `dst`: canonical allowed node id.
- `relation`: one enum value from the list above.
- `weight`: normalized numeric edge strength.
- `confidence`: source confidence before graph weighting.
- `stage`: `deterministic`, `semantic`, `evidence`, or `mixed`.
- `sources`: list of producer/source ids such as `clang_text_spans`, `clang_preprocess`, `text_fallback`, or provider id.
- `attr`: object with required `source` list plus optional `fields`, `resolver_wrappers`, `provider`, `model`, `job_id`, and `evidence`.

Relation normalization examples:

- `field_set`, `field_write`, `REG_SET_FIELD`, and LLM `sets field` normalize to `sets_field`.
- `has_field` does not become a product edge by itself because fields are attributes. It can contribute to `register.attr.fields`.
- `contains_box` normalizes to `contains`.
- `documents_register`, `documented_by`, `section_mentions`, and provider wording such as `explains` normalize to `documents`.
- Unrecognized semantic provider relations normalize to `relates_to` and keep the original value in `attr.original_relation`.
- Wrapper relations such as `wraps` are provenance only and must not become product graph edges.

## Edge Provenance Contract

| Layer | `edges.stage` | `edges.source` | Producer | Required provenance | Default graph role |
| --- | --- | --- | --- | --- | --- |
| Stage 1 deterministic code graph | `deterministic` | `clang_text_spans`, `clang_preprocess`, `clang_callback`, or `text_fallback` | `packages/core/src/asip/code_graph.py` during indexing | extractor, function, resolver profile, wrapper, symbol argument, analysis mode, path/line; callback edges additionally record slot, receiver, callback table, optional `type_flow=clang_ast_json`, and optional `callback_initializer_flow=clang_ast_json` | Included in default graph |
| Evidence/query overlay | `evidence` | `query_expected_terms` or derived evidence overlay | indexed evidence rows and legacy query-term fixtures | extractor, query/config marker, source path/line or section | Only included when evidence overlay is explicitly requested or when no persisted Stage 1/2 graph exists and the DB is below the evidence row cap |
| Stage 2 semantic overlay | `semantic` | provider id such as `ollama` | semantic-edge query or batch jobs | provider, model, job id, mode, source evidence/candidate | Included in default graph |

The default product graph must not treat evidence/query-term edges as Stage 1. Evidence overlay is a debug/context layer, not proof that function/register extraction is complete.

## Remaining Gap

The graph API is live-data backed and now supports a no-seed global graph plus explicit query-scoped and batch semantic-edge actions. The storage contract separates Stage 1 deterministic, evidence overlay, and Stage 2 semantic edges, and the current Stage 1 tests prove function/register operation edges, cross-file direct common-helper calls when the callee definition is unique, plus direct helper and ops/vtable callback `calls` edges with fields, wrappers, callback slots, and source paths folded into BoxMatrix-style `in`/`out`/`attr` payloads.

The entity-contract gap is no longer blocking for the current product graph: field, macro/wrapper, callback slot, source-file, and control-keyword pseudo nodes are filtered or folded, edge relations are enum-normalized, and the UI refuses legacy `field/doc/pdf/code` graph-node kinds. Remaining follow-up risk is narrower but still real: the Stage 1 extractor is still pragmatic clang/text-span extraction plus conservative direct-call/callback dispatch expansion rather than full clangd/libclang cursor traversal across every Linux kernel compile-database and callback edge case. Ambiguous duplicate cross-file callees are intentionally skipped instead of over-connected until a typed call graph is available, and Stage 2 larger semantic batches still need local-model JSON robustness testing.

The final product still needs continued visual QA as graph density grows, but `/graph` and query-scoped graph panels now render indexed ASIP graph output rather than fixed page rows, fixed edge counts, or hardcoded layout assumptions.

## Acceptance Criteria

- Graph API reads persisted graph data or core graph runtime output.
- `/graph` renders the full indexed weighted relation graph, while query/inspector views can render bounded 1-2 hop subgraphs.
- The `/graph` route must be treated as an active product feature, not a post-MVP visual-only deferral.
- `/graph` uses a maintained React/npm graph package; hand-written SVG graph layout is not acceptable for completion.
- Users can pan/zoom/inspect the global graph, and the default viewport shows a recognizable global relation graph rather than a tiny fixed seed diagram.
- Stage 1 deterministic graph build uses a code-aware preprocessor/macro-expansion path plus selective clang AST JSON receiver/initializer hints for C/C++ corpora and records graph-build provenance. Full clangd/libclang cursor traversal remains a residual unless separately implemented and tested.
- The default global graph includes Stage 1 function nodes and operation edges that connect code functions to the registers they read, write, or set; resolver macro/wrapper names, expanded macro helpers, and field-only symbols are not entity nodes.
- Stage 1 deterministic graph build links direct helper calls and ops/vtable callback dispatch into function-to-function `calls` edges, so common dispatch logic can connect to callback functions and then to the registers those callbacks operate on.
- Graph node payloads expose BoxMatrix-style `in`, `out`, and `attr` fields, with register fields, wrapper names, source path/line, and doc summaries carried as attributes/provenance.
- Converted Markdown/PDF document sections become graph nodes with source/page/heading provenance, and section-to-symbol/semantic edges are visible in graph output.
- Stage 2 batch LLM semantic-edge generation can process indexed corpus candidates, persist separately identifiable semantic edges, and refresh graph output without requiring a single selected query seed.
- The UI exposes graph size and minimum edge-weight controls instead of silently truncating nodes/edges in the component.
- Edge width, opacity, or emphasis reflects edge weight/confidence.
- Querying changes graph nodes and edges.
- The graph remains visible and legible in light and dark themes.
- NetworkX-derived traversal/subgraph output is used by the product graph API, or this is explicitly accepted as a post-MVP deferral.
